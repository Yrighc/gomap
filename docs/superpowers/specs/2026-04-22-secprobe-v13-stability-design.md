# GoMap secprobe v1.3 单目标稳定性与结果表达增强设计

日期：2026-04-22

## 1. 背景

截至 `v1.2`，GoMap `secprobe` 已经完成从“账号口令尝试”到“统一弱认证探测层”的第一轮升级：

- 支持 `credential` / `unauthorized` 双类型探测
- 已接入 Redis / MongoDB 未授权访问探测
- 支持成功 finding 后的可选 enrichment
- 保持 `gomap weak` / `gomap port -weak` 统一入口

这意味着 `v1.3` 不再需要回头解决“统一入口是否成立”或“未授权协议是否应该进入 `secprobe`”这类基础问题。

当前更真实的改进方向是：

- 单目标流程还需要更稳
- finding 结果还需要更准
- 执行状态和失败原因还需要更清晰

同时，`v1.3` 也已经明确不应被做成：

- 批量并发治理版本
- 大规模扩协议版本
- CLI 参数快速膨胀版本
- 平台化控制层版本

因此，`v1.3` 的核心定位应当是：在保持当前统一入口和默认保守行为不变的前提下，把单目标认证探测链路做稳、做准、做得更容易被后续扩展和验证。

## 2. 目标与非目标

### 2.1 目标

- 围绕 `pkg/secprobe` 完成 `v1.3` 单目标稳定性增强设计
- 增强公共执行骨架，让单目标探测流程具备更清晰的阶段状态和失败分类
- 增强内部结果模型，使其能表达：
  - 命中
  - 跳过
  - 失败
  - 确认不足
  - enrichment 补采状态
- 在不明显增加 CLI 理解成本的前提下，保持对外入口可用性和兼容性
- 对现有高价值协议做定点确认逻辑加固，优先减少误判

### 2.2 非目标

- 本次不新增大量新协议
- 本次不引入多目标并发治理
- 本次不引入批量喷洒防护、全局速率限制、批量任务保护策略
- 本次不做大规模 CLI 选项扩展
- 本次不把内部结果模型直接稳定成最终对外输出契约
- 本次不引入平台化任务编排、状态中心、外部存储
- 本次不回退 `assetprobe` / `secprobe` 当前边界

## 3. 方案对比

### 方案 A：只补公共骨架和结果模型，协议侧最小适配

做法：

- 主要增强 `SecurityResult` 与执行骨架
- 协议侧只做必要兼容性适配

优点：

- 风险最小
- 改动边界最收敛

缺点：

- 对“误判控制优先”支持不足
- 用户感知收益可能偏弱

### 方案 B：公共骨架增强 + 内部结果模型升级 + 高价值协议定点加固

做法：

- 增强 `Run` 执行链路
- 给 `SecurityResult` 增加内部结构化语义字段
- 只对少数高价值协议补确认逻辑和失败分类

优点：

- 与 `v1.3` 目标最一致
- 能直接提升单目标探测质量
- 不会把范围扩散成全面重写

缺点：

- 需要谨慎处理内部结果字段与外部兼容边界
- 需要补更细的测试断言

### 方案 C：现有协议全面回炉，统一补确认、重试和证据标准

做法：

- 对所有 credential / unauthorized 协议统一做深度改造

优点：

- 一致性最强

缺点：

- 范围明显超出 `v1.3`
- 风险、验证成本和回归面都偏大

### 结论

选择方案 B。

即：

- `v1.3` 先增强公共骨架和内部结果模型
- 协议侧只做高价值定点加固
- 不做全协议重构

## 4. 总体设计

### 4.1 模块边界

`v1.3` 继续保持当前分层，不新增新模块，也不改变统一入口：

- `pkg/secprobe`
  - 统一对外入口
  - 执行编排
  - 默认策略
  - 结果归一化与收口
- `internal/secprobe/core`
  - 公共模型
  - 注册器
  - 探测器接口约定
- `internal/secprobe/<protocol>`
  - 单协议确认逻辑
  - 单协议证据生成
  - 单协议错误分类辅助

边界原则：

- 公共层负责流程语义和结果收口
- 协议层负责最小但可靠的确认动作
- 不让协议层反向决定全局 stop policy 和对外输出策略

### 4.2 执行链路定位

`v1.3` 的重点不是增加更多尝试，而是把单目标执行链拆清楚。

建议把单目标探测统一理解为以下四段：

1. `matched`
   - 候选命中了某协议和某类 `ProbeKind`
2. `attempted`
   - 已发起实际认证尝试或未授权验证
3. `confirmed`
   - 已通过协议确认逻辑，结果可以作为有效 finding
4. `enriched`
   - 已对成功 finding 完成补采

说明：

- 这些状态首先服务于内部执行语义和测试断言
- `Stage` 属于内部增强字段，不要求 CLI 在 `v1.3` 做大规模显式展示

### 4.3 确认原则

`v1.3` 明确一条总原则：

只有协议确认成立，才能把结果记为成功。

因此：

- credential 类协议
  - 不能只因为建立连接或完成握手就算成功
  - 必须有明确认证成功信号
- unauthorized 类协议
  - 不能只因为端口可连或接口有响应就算成功
  - 必须通过最小确认动作证明“无需认证即可访问有效信息”

## 5. 结果模型设计

### 5.1 总体原则

`v1.3` 不引入第二套结果模型，而是在现有 `SecurityResult` 上做兼容式增强。

增强原则：

- 保留现有对外兼容字段
- 新增内部结构化字段
- CLI 继续优先沿用现有展示逻辑
- 新字段先服务于内部执行、测试和后续扩展

### 5.2 兼容保留字段

以下字段继续保留，并保持现有对外语义：

- `Target`
- `ResolvedIP`
- `Port`
- `Service`
- `ProbeKind`
- `FindingType`
- `Success`
- `Username`
- `Password`
- `Evidence`
- `Enrichment`
- `Error`

### 5.3 建议新增内部字段

建议为 `SecurityResult` 引入以下内部结构化字段：

- `Stage`
  - 表达执行阶段
  - 建议取值：
    - `matched`
    - `attempted`
    - `confirmed`
    - `enriched`
- `SkipReason`
  - 表达跳过语义
  - 例如：
    - 协议不支持
    - 对应 `ProbeKind` 未启用
    - 无可用凭证
    - 候选不匹配
- `FailureReason`
  - 表达失败语义
  - 例如：
    - 连接失败
    - 认证失败
    - 超时
    - 上下文取消
    - 确认不足
- `Capabilities`
  - 仅做最小表达
  - 主要为 unauthorized / enrichment 场景预留
  - `v1.3` 不做复杂分级
- `Risk`
  - 先作为内部增强字段
  - 用于后续更稳定的输出契约演进

### 5.4 字段取舍说明

`Stage` 的价值较高，应在 `v1.3` 正式落地。

原因：

- 可直接服务于单目标流程稳定性
- 能减少对 `Error` / `Evidence` 文本的语义猜测
- 能让测试断言从“看字符串”过渡到“看状态”

`Capabilities` 建议弱化处理，只保留最小扩展位。

原因：

- 它对 unauthorized 结果分层有价值
- 但当前主线不是复杂风险分级，而是流程跑稳和结果做准

## 6. 执行流程设计

### 6.1 流程职责拆分

公共执行链负责：

- 选择 `ProbeKind`
- 构造默认结果
- 管理 `Stage` 推进
- 归一化 `SkipReason` / `FailureReason`
- 控制目标级熔断边界
- 控制 stop policy
- 控制有限重试边界
- 控制 enrichment 触发时机

协议探测器负责：

- 进行协议级探测
- 执行最小确认动作
- 生成协议级证据
- 尽量避免基于模糊响应直接判成功

### 6.2 单目标稳定性增强点

建议在公共执行链补齐以下能力：

- 更清晰的 skip / failure 分类
- 对“已尝试但未确认”的结果单独归类
- 目标级熔断与提前终止边界更明确
- 限定少量明确场景下的有限重试
- 将 stop policy 作用边界明确到单目标执行链
- 增强链路可观测性，便于测试和后续审计

### 6.3 重试策略

`v1.3` 不建议引入通用多次重试框架。

建议策略：

- 默认不做大范围重试
- 只在少数明确、可解释的临时失败场景下做有限重试
- 重试的目的不是增加撞库次数，而是降低瞬时网络抖动对结果的污染

### 6.4 CLI 控制面原则

`v1.3` 不建议继续堆很多显式控制选项。

控制面原则应保持：

- 默认行为保守
- 开关语义直观
- 少量高价值参数即可覆盖主场景
- 尽量复用现有参数语义

如果确实需要新增控制项，也只应增加与单目标稳定性直接相关且用户容易理解的少量参数。

## 7. 协议加固范围

### 7.1 总体原则

`v1.3` 不做全协议回炉，而是按“误判风险 + 当前价值”选择少量协议做定点增强。

### 7.2 优先协议

优先建议加固：

- Redis unauthorized
- MongoDB unauthorized

原因：

- 它们是 `v1.2` 新接入能力
- 当前最适合作为 unauthorized 结果语义和确认逻辑的基线样板

### 7.3 Credential 协议加固策略

建议再选择 1 到 2 个高价值 credential 协议做定点加固。

选择原则：

- 使用频率高
- 易受确认逻辑不充分影响
- 当前失败分类或证据表达容易混淆

本设计不在 spec 中预先锁定具体协议名称，但实施计划阶段必须从现有 credential 协议中明确选出 1 到 2 个目标协议，并说明选择依据。

### 7.4 协议确认要求

每个被加固协议都应满足：

- 有明确的“确认成功”条件
- 有明确的“确认不足”落点
- 返回证据可被测试断言
- 不依赖外部平台语义或复杂审计上下文

## 8. 测试与验证设计

### 8.1 测试目标

`v1.3` 的测试重点不只是看 `Success` 布尔值，而是验证流程语义是否正确。

建议覆盖：

- `Stage` 推进是否符合预期
- `SkipReason` 与 `FailureReason` 是否可区分
- 协议是否满足“确认后才能成功”
- enrichment 失败是否不影响主 finding 成败
- CLI 在内部模型增强后是否仍保持兼容与易读

### 8.2 测试层次

建议按三层组织：

- 核心模型与执行链单测
  - 校验默认结果、阶段推进、失败归类
- 协议级单测 / 集成测试
  - 校验高价值协议的确认逻辑
- CLI 回归测试
  - 校验对外行为不被内部模型增强破坏

### 8.3 验证原则

验证时继续保持：

- 先有测试再落实现
- 分阶段验证
- 每个阶段都有独立可回归结果

## 9. 实施拆分建议

建议 `v1.3` 按四个子阶段推进：

### 9.1 `v1.3-a` 公共结果模型增强

目标：

- 为 `SecurityResult` 增加内部结构化字段
- 明确 `Stage / SkipReason / FailureReason`
- `Capabilities` 只做最小保留

### 9.2 `v1.3-b` 公共执行链增强

目标：

- 在 `Run` / 归一化链路补阶段推进
- 统一 skip / failure 语义
- 明确 stop policy 与有限重试边界

### 9.3 `v1.3-c` 高价值协议定点加固

目标：

- 优先补 Redis / MongoDB unauthorized
- 再补 1 到 2 个高价值 credential 协议
- 每个协议都有可测试的最小确认动作

### 9.4 `v1.3-d` 文档与兼容性收口

目标：

- 更新 README / 示例 / CLI 文档
- 说明哪些结果语义属于内部增强
- 跑完整体验证并完成 review

## 10. 风险与约束

### 10.1 主要风险

- 结果模型容易被一次性设计得过重
- 协议加固范围容易在实施时不断扩张
- 内部增强字段若过早暴露，可能对外形成不必要承诺

### 10.2 约束原则

- 不为未来 `v2.0` 过度预埋复杂模型
- 不让 `v1.3` 变成“隐性扩协议版本”
- 不让 CLI 为内部模型增强承担过高复杂度

## 11. 完成标准

当以下条件满足时，可认为 `v1.3` 设计目标达成：

- 单目标执行链具备明确阶段语义
- finding 的跳过、失败、确认成功、补采状态可被结构化表达
- 高价值协议的确认逻辑得到定点加固
- CLI 入口保持易用，没有明显增加理解成本
- 不破坏 `v1.2` 已形成的统一入口和默认保守边界

## 12. 结论

`secprobe v1.3` 的价值不在于“再多接几个协议”，而在于把单目标认证探测这条链路做成更稳、更准、更容易验证的能力底座。

这一版最合理的路径是：

- 公共结果模型增强
- 公共执行链增强
- 少量高价值协议定点加固
- 保持 CLI 简洁和外部兼容

这样既能承接 `v1.2` 已完成的统一弱认证骨架，也能为 `v2.0` 的稳定输出契约和可扩展能力打下更合适的基础。
