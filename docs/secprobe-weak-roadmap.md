# GoMap 弱口令探测后续优化升级 Roadmap

日期：2026-04-21

## 1. 文档目的

本文用于沉淀当前 `secprobe` / `weak` 能力在 `v1.0` 落地后的后续升级路线，方便后续按阶段拆分 spec、plan 和实现任务。

本文关注的是：

- 当前 `v1.0` 已交付能力的真实边界
- 后续版本的建议演进方向
- 每个阶段的目标、范围、关键改动点、风险
- 推荐实施顺序，避免后续演进重新把 `assetprobe` 与 `secprobe` 职责搅混

## 2. 当前 v1.0 基线

### 2.1 已实现能力

当前仓库已经具备以下 `v1.0` 弱口令探测能力：

- 独立协议安全探测层：`pkg/secprobe`
- 独立命令：`gomap weak`
- 串联命令：`gomap port -weak`
- 支持协议：
  - `ssh`
  - `ftp`
  - `mysql`
  - `postgresql`
  - `redis`
  - `telnet`
- 支持内置协议字典和自定义字典目录
- 支持内联凭证与文件凭证输入
- 支持按协议过滤、并发控制、超时控制、命中即停
- `port -weak` 输出使用最小包裹结构：
  - `asset`
  - `security`

### 2.2 当前边界

`v1.0` 的设计边界已经明确，不建议后续回退：

- `pkg/assetprobe` 继续只负责资产发现
- `pkg/secprobe` 负责协议安全探测
- 不把弱口令结果重新写回 `assetprobe.PortResult`
- `port -weak` 必须显式开启，不能恢复历史上的“顺手爆破”
- `port -weak` 当前只支持 `tcp`
- 当前 `FindingType` 实际只覆盖 `credential-valid`

### 2.3 当前短板

和后续演进直接相关的短板主要有：

- 探测类型仍然只覆盖“账号口令尝试”
- 缺少 `redis` / `mongodb` 这类未授权访问能力
- 协议候选识别仍然偏轻量，复杂服务名兼容还不够多
- 缺少速率治理、目标级熔断、喷洒保护等更稳健的执行策略
- 结果模型对“发现类型”“证据结构”“风险分级”表达还不够完整
- 缺少更适合平台接入的结果输出、审计与复用能力

## 3. 演进原则

后续所有版本建议继续遵守以下原则：

### 3.1 保持模块边界稳定

- `assetprobe` 不负责安全验证
- `secprobe` 不负责全量端口发现
- 不让 `cmd/main.go` 继续膨胀为一层复杂调度器

### 3.2 先做“能力扩展”，再做“平台化”

建议先补齐协议能力与探测类型，再考虑复杂审计、任务编排、资产平台对接。

### 3.3 先抽象结果模型，再扩未授权协议

如果直接往现有 `credential-valid` 上堆 `redis unauth`、`mongodb unauth`，很容易把结果结构做乱。更合理的顺序是：

1. 先扩展 `FindingType` 与证据模型
2. 再接入未授权访问协议

### 3.4 默认保守，显式开启

后续就算加入更强能力，也建议继续保持：

- 默认不开启
- 默认限速
- 默认最小必要尝试
- 明确区分“认证尝试”和“未授权访问验证”

## 4. 总体版本路线

建议按四个阶段推进：

| 版本 | 主题 | 核心目标 |
| --- | --- | --- |
| `v1.1` | 稳定性与模型补强 | 先把当前能力做稳、做清晰 |
| `v1.2` | 未授权访问接入 | 引入 `redis` / `mongodb` 未授权访问 |
| `v1.3` | 执行策略升级 | 补齐治理、熔断、速率和更细的控制面 |
| `v2.0` | 平台化与扩展层 | 让 `secprobe` 成为稳定的协议安全探测层 |

## 5. v1.1：稳定性与模型补强

### 5.1 目标

在不扩大协议范围的前提下，把当前 `v1.0` 打磨成可持续扩展的基础版本。

### 5.2 建议范围

- 扩展 `FindingType`
- 补齐统一结果分类
- 优化候选识别与服务归一化
- 完善错误模型和统计信息
- 补齐更稳定的测试和文档

### 5.3 关键改动点

#### 5.3.1 扩展 `FindingType`

建议从当前单一的 `credential-valid` 扩展为：

- `credential-valid`
- `credential-invalid`
- `credential-error`
- `unauthorized-access`
- `protocol-matched-but-skipped`

这里不是要求所有类型立刻都在 CLI 中输出，而是先把公共模型定下来。

#### 5.3.2 引入更清晰的证据结构

当前 `Evidence` 更适合字符串。后续建议演进为“简化证据对象”，例如：

- 认证成功方式
- 返回特征摘要
- 关键服务响应
- 是否只读、是否可执行写操作

`v1.1` 不一定要完全对象化，但至少需要把后续演进路径定清楚。

#### 5.3.3 完善服务归一化规则

重点补强：

- `redis/tls`
- `pgsql`
- `mysql?`
- `ssh?`
- 更复杂 banner 下的协议归并

#### 5.3.4 统一跳过原因

当前的“未匹配”“协议不支持”“无凭证”“候选为空”建议收敛成结构化 skip reason，便于后续平台消费。

### 5.4 风险

- 容易为了结果表达去过度设计模型
- 容易把 `v1.1` 误做成“开始支持很多新协议”

### 5.5 完成标准

- 不新增大范围协议实现
- 结果模型能兼容后续未授权访问
- 候选、失败、跳过、成功都有明确统计和表达
- 文档和示例与真实 CLI 保持一致

## 6. v1.2：未授权访问接入

### 6.1 目标

在 `secprobe` 仍保持“协议安全探测层”定位的前提下，增加最有价值的未授权访问能力。

### 6.2 建议优先级

优先建议：

1. `redis` 未授权访问
2. `mongodb` 未授权访问

暂缓建议：

- `elasticsearch`
- `memcached`
- `zookeeper`
- `kafka`

原因是 `redis` / `mongodb` 与当前讨论目标最一致，也最能验证结果模型是否合理。

### 6.3 设计建议

#### 6.3.1 不要强塞进现有 credential probe

建议把探测类型从“只有 credential”升级成“安全探测模式”，例如：

- credential attempt
- unauthorized access check

不要把未授权访问伪装成“用户名为空、密码为空”的认证尝试。

#### 6.3.2 建议补一层 probe kind

例如在内部执行路径区分：

- `ProbeKindCredential`
- `ProbeKindUnauthorized`

这样后续 CLI、结果模型、审计都更容易区分。

#### 6.3.3 未授权访问要分级

对于未授权访问，建议至少区分：

- 仅连通但未确认
- 已确认未授权访问
- 已确认可读
- 已确认可写

`v1.2` 可以先做到“是否确认未授权”与“只读/读写”的粗粒度。

### 6.4 CLI 建议

建议新增显式开关，而不是默认随着 `weak` 一起打开，例如：

- `gomap weak -enable-unauth`
- `gomap port -weak -enable-unauth`

也可以考虑按协议类型显式指定：

- `-modes credential,unauth`

如果实现想先简单一些，推荐优先使用单开关模式。

### 6.5 风险

- 未授权访问的“确认逻辑”比认证成功更容易误报
- 结果表达不清楚时，使用方会混淆“认证成功”和“匿名可访问”
- 一旦直接支持写操作验证，风险会明显上升

### 6.6 完成标准

- `redis` 未授权访问可稳定验证
- `mongodb` 未授权访问可稳定验证
- 结果中明确区分 `credential-valid` 与 `unauthorized-access`
- CLI 参数可以显式控制是否启用未授权访问探测

## 7. v1.3：执行策略升级

### 7.1 目标

让 `secprobe` 从“能跑”升级到“更稳、更可控、更适合批量使用”。

### 7.2 建议范围

- 目标级熔断
- 协议级限流
- 全局速率限制
- 更细的 stop policy
- 字典策略增强
- 批量目标保护策略

### 7.3 关键改动点

#### 7.3.1 目标级熔断

例如：

- 单目标连续失败达到阈值后停止该目标
- 单协议对同一目标达到最大尝试数后停止

#### 7.3.2 协议级并发与速率

当前并发更偏全局视角。后续建议细化为：

- 全局并发
- 每协议并发
- 每目标并发
- 每秒尝试数限制

#### 7.3.3 Stop policy 细化

当前主要是 `stop-on-success`。后续可细化为：

- 单协议命中后停止
- 单目标任一协议命中后停止
- 单目标达到阈值后停止
- 全局达到命中数后停止

#### 7.3.4 字典策略增强

建议优先做“简单但有价值”的增强：

- 用户名字典与密码字典分离组合
- 按协议加载更精简默认字典
- 支持去重与优先级顺序
- 支持“小字典优先，大字典后置”

不建议过早引入复杂 DSL。

### 7.4 风险

- 调度策略一复杂，`run.go` 很容易失控
- CLI 参数可能快速膨胀

### 7.5 完成标准

- 大批量目标运行时更稳定
- CLI 虽然能力增强，但仍保持可理解
- 调度复杂度主要收敛在 `secprobe` 内部，而不是散落到 CLI

## 8. v2.0：平台化与扩展层

### 8.1 目标

把 `secprobe` 从“当前 GoMap 的一个附加能力”升级成稳定的协议安全探测层，方便被外部系统复用。

### 8.2 建议方向

- 更稳定的公共结果模型
- 更明确的探测任务对象
- 多种输出适配
- 审计/留痕能力
- 插件化协议注册

### 8.3 可能包含的能力

#### 8.3.1 任务对象化

从当前偏函数式调用，演进为更明确的任务对象，例如：

- 任务 ID
- 目标批次
- 探测模式
- 执行策略
- 输出控制

#### 8.3.2 输出层扩展

建议支持：

- JSON
- JSONL
- CSV 摘要
- 便于平台接入的结构化 summary

#### 8.3.3 协议注册扩展

如果后面确实还要增加更多协议，建议在 `v2.0` 再考虑：

- 更完整的 registry metadata
- capability 声明
- 插件式加载

而不是在 `v1.x` 就提早做成通用漏洞框架。

### 8.4 不建议在 v2.0 之前做的事

- 一步到位变成通用 GoPoc 引擎
- 同时支持几十个协议
- 复杂规则 DSL
- 全量漏洞利用或利用链能力

### 8.5 完成标准

- `secprobe` 作为包可独立被业务系统消费
- 输出层足够稳定
- 扩展协议时不需要大改主流程

## 9. 推荐实施顺序

建议按下面顺序推进，而不是按“先上更多协议”推进：

1. `v1.1`：先稳住模型和结果表达
2. `v1.2`：再接 `redis` / `mongodb` 未授权
3. `v1.3`：补执行治理与控制面
4. `v2.0`：最后再做平台化与扩展层

原因：

- 先扩模型，后扩能力，返工最少
- 先把探测类型分清，后面未授权访问才不会做乱
- 先做内部边界稳定，再谈更多输出和对外复用

## 10. 近期优先事项建议

如果后面只开一轮迭代，建议优先做下面三件事：

### 10.1 第一优先级

补齐 `v1.1` 中的结果模型与 `FindingType` 预留。

### 10.2 第二优先级

落 `redis` 未授权访问，作为首个非账号口令类型探测。

### 10.3 第三优先级

落 `mongodb` 未授权访问，并验证模型是否还能保持清晰。

## 10.4 基于 Chujiu_reload 的 v1.2 修正

参考 `/Users/yrighc/work/hzyz/project/Chujiu_reload` 后，建议把 `v1.2` 从“只增加两个未授权协议”收敛为“统一弱认证能力升级”：

- 保留 `secprobe` 统一入口
- 内部明确区分：
  - `credential`
  - `unauthorized`
- 首批增加：
  - `redis` 未授权访问
  - `mongodb` 未授权访问
- 新增可选 enrichment，用于命中后详情补采

这里借鉴的是它的能力拆分与成功后补采思路，不借鉴其平台编排与存储层。

## 11. 后续拆分建议

后续如果继续做，可以按下面顺序拆成独立 spec：

1. `secprobe v1.1 结果模型与执行状态补强`
2. `secprobe v1.2 redis unauth`
3. `secprobe v1.2 mongodb unauth`
4. `secprobe v1.3 执行策略升级`
5. `secprobe v2.0 输出层与平台接入`

## 12. 结论

当前 `v1.0` 已经完成了“把弱口令从历史遗留钩子中重新抽离，并以独立安全探测层落地”的关键一步。

后续最重要的不是立刻堆更多协议，而是：

- 先把结果模型做稳
- 再把未授权访问接进来
- 再把执行策略做强
- 最后再考虑平台化和更大范围扩展

如果后续继续按这个顺序推进，`secprobe` 会更像一个边界清晰、可持续演进的协议安全探测层，而不是重新滑回“大一统扫描器”。
