# GoMap v0.4.7
> secprobe 能力扩展、协议元数据治理收口、共享弱口令字典升级、enrichment 首批落地

---

## 概览

本版本是相对 `v0.4.6` 的一次 `secprobe` 主题增强发布，重点围绕 **协议支持面扩展**、**协议元数据驱动执行**、**共享弱口令字典治理**、**原子认证与未授权执行路径统一**、以及 **认证成功后的 enrichment 补充信息** 展开。

这一轮变化的核心不是单点补几个协议，而是把 `secprobe` 的默认执行模型、协议声明方式、字典来源、失败分类和结果补充能力继续收敛为一条更稳定的主路径：

`metadata -> planner -> engine -> atomic/unauthorized provider -> enrichment`

同时，弱口令候选生成进一步统一到：

`metadata.dictionary -> credential source -> generator -> engine`

这意味着后续不管是新增协议、扩展默认字典、补充默认 registry、还是在平台侧消费结果结构，都有了更清晰的边界和更一致的行为。

**对比基线**：`v0.4.6`

**更新规模**：143 commits | 205 files changed | +44,462 / -736 lines

---

## 重点内容

- **secprobe 协议支持面继续扩展**：补齐多批 credential / unauthorized / http-api 登录型协议能力
- **协议元数据目录化成为主心智**：内置协议声明继续集中到 `app/secprobe/protocols/*.yaml`
- **共享弱口令字典治理落地**：默认 builtin 弱口令来源切换为共享密码池 + 协议差异 metadata
- **atomic credential 执行路径继续收口**：内置协议进一步从 legacy batch 模式迁移到单次原子认证模型
- **新增第一批 enrichment 能力**：认证成功后可追加轻量证明信息，当前覆盖 `postgresql`、`mysql`、`elasticsearch`
- **弱口令扫描体验优化**：默认 timeout 调整为 `1s`，并修复“自定义端口 + 显式协议”场景下 candidate 丢失的问题
- **未授权探测路径完善**：`memcached`、`zookeeper` 已接入并补齐失败分类与注册顺序边界
- **资源目录与嵌入资源治理增强**：`app` 目录按扫描引擎拆分布局，资源校验测试同步补齐

## 新功能

### secprobe 协议能力扩展

本版本新增或补齐了以下协议能力：

- Credential：
  - `mssql`
  - `rdp`
  - `vnc`
  - `smb`
  - `smtp`
  - `amqp`
  - `oracle`
  - `snmp`
  - `mongodb`
  - `imap`
  - `pop3`
  - `kafka`
  - `ldap`
  - `activemq`
  - `zabbix`
  - `neo4j`
  - `elasticsearch`
- Unauthorized：
  - `memcached`
  - `zookeeper`
- Enrichment：
  - `postgresql`
  - `mysql`
  - `elasticsearch`

### enrichment 首批落地

新增认证成功后的结果补充能力，当前会在原 finding 语义不变的前提下，向 `enrichment` 字段追加轻量证明信息：

- `postgresql`：执行 `SELECT version();`
- `mysql`：执行 `SELECT @@version;`
- `elasticsearch`：访问 `GET /_security/_authenticate`

平台侧可以直接消费 `enrichment.payload` 作为成功后的附加证明文本。

### 协议元数据与目录化治理

- 内置协议静态声明继续集中到 `app/secprobe/protocols/*.yaml`
- 新增或补齐多个协议的：
  - 端口
  - alias
  - capability
  - 默认用户名
  - 密码来源
  - 结果语义
  - enrichment 支持开关

### 共享密码池与凭证生成能力

- 默认 builtin 弱口令来源继续切换到共享密码池
- 支持：
  - 分层档位
  - 默认用户
  - 协议额外密码
  - 默认用户名密码对
  - 基础变异规则
  - 显式凭证语义收紧

### HTTP/API 登录型认证复用

- 新增 `internal/secprobe/httpauth` 复用层
- 当前已用于：
  - `zabbix`
  - `neo4j`

## 变更

### secprobe 执行模型进一步统一

- 默认执行路径继续收口到 `metadata -> planner -> engine -> provider`
- 内置协议的 `credential` 能力进一步切换到 atomic provider 为中心
- legacy public prober 仍保留兼容入口，但不再是内置协议主实现模式

### 资源目录结构调整

- `app` 目录按 `assetprobe` / `secprobe` 拆分资源布局
- 服务映射、服务探针、目录字典与 secprobe 资源入口继续治理收口
- embed 资源加载与校验测试同步补齐

### 弱口令运行时默认行为调整

- `weak` 模式默认 `timeout` 收敛为 `1s`
- 适合内网批量扫描场景，减少对单目标长时间等待
- 对公网或高延迟环境，如有需要可显式传入更大 `-timeout`

### candidate 构造行为增强

- 当端口没有识别出服务指纹，但用户显式传入 `-protocols` 时，仍会按显式协议构造候选
- 例如：
  - `weak -ports 10033 -protocols ssh`
  - 即使没有指纹识别出 `ssh`，也能进入 secprobe 探测

### PostgreSQL 默认用户名扩展

- `postgresql` 默认用户名枚举新增 `root`
- 用于覆盖更多内网弱口令场景下的实际用户名分布

## 移除

### 旧式协议独立字典主路径的推荐地位

- 不再建议继续维护“每协议一份独立密码库”作为默认 builtin 主模型
- 默认 builtin 弱口令来源已切换为：
  - 共享密码池
  - 协议差异 metadata

### legacy 执行模型的主路径地位

- legacy public `Prober` 仍保留兼容能力
- 但不再代表当前内置协议推荐扩展方式

## 修复

### 协议实现与失败分类

- 修复 `elasticsearch` 证书错误后的 insecure TLS retry，使其真正作用于 transport，而不是仅停留在测试标记
- 修复 `elasticsearch` 的 HTTPS / HTTP 回退边界与 IPv6 URL 组装问题
- 修复 `smtp` 认证回退与 implicit TLS timeout 保护
- 修复 `oracle`、`snmp`、`rdp`、`mongodb`、`vnc`、`smb` 等协议在第一版接入后暴露出的端口边界、超时透传、错误分类或成功判定问题

### 候选生成与字典装载

- 修复候选过滤与 alias 字典加载回归
- 修复 `ResolvedIP` 缺失时错误透传空主机的问题
- 修复“自定义端口 + 显式协议”场景下 candidate 漏构造的问题
- 收紧显式凭证的字面语义，避免被隐式扩展污染

### 未授权与模板执行路径

- 收紧 `memcached` 未授权成功判定与断连分类
- 修复 `zookeeper` 默认注册顺序、会话状态判定与失败语义
- 完成 `memcached` 未授权模板执行器接入并收紧边界

### 测试与夹具稳定性

- 修复 testcontainers 返回 `localhost / ::1` 时导致的服务探测测试不稳定
- 收紧 MySQL、MongoDB 等夹具就绪判定
- 默认继续跳过部分服务探测集成测试，降低 CI 波动

## 说明与注意事项

- **这是一次 secprobe 主路径持续收口版本**：如果你只是调用 `Run` / `RunWithRegistry`，多数 public API 不需要改，但需要接受新的内部执行与声明模型。
- **默认弱口令维护模型已改变**：后续不建议继续为每个协议复制整份密码库，优先维护共享密码池与协议差异 metadata。
- **enrichment 目前只覆盖第一批协议**：当前为 `postgresql`、`mysql`、`elasticsearch`，其他协议暂未补齐。
- **HTTP/API 登录型认证仍是 provider 层复用能力**：`httpauth` 不是公共 DSL，也不是第二套 engine。
- **未授权模板执行器仍是受限模型**：适合简单确认，不适合多阶段状态机协议。

## 升级与集成

### 作为 Go 依赖升级

如果你以库形式使用 `gomap`，建议：

```bash
go get github.com/yrighc/gomap@v0.4.7
go mod tidy
```

升级后建议重点检查：

- 是否依赖旧的协议独立默认字典行为
- 是否假定弱口令默认 timeout 仍是 `3s` 或 `5s`
- 是否需要在平台结果展示中接入 `enrichment` 字段

### 作为 CLI 使用

- 内网批量扫描场景可直接受益于默认 `1s` timeout
- 若目标是公网、跨地域或高延迟资产，请显式增大 `-timeout`
- 使用自定义端口时，建议显式传入 `-protocols`，这样在无指纹场景下也能进入 secprobe 主路径

### 对三方协议扩展方

- 新协议扩展优先遵循：
  - `metadata`
  - `planner`
  - `engine`
  - `atomic / unauthorized provider`
- 不建议继续以 legacy batch prober 作为新的主实现模式
