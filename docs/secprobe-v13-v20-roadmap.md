# GoMap secprobe 后续版本路线（v1.3 / v1.4 / v2.0）

日期：2026-04-22

## 1. 文档目的

本文用于在 `secprobe v1.2` 已交付的基础上，明确下一阶段的演进方向，方便后续继续拆分：

- spec
- implementation plan
- 迭代任务

本文重点关注：

- `v1.3` 应该先补什么
- `v1.4` 应该先修什么
- `v2.0` 适合承接什么
- 哪些能力值得继续做
- 哪些能力当前不建议提前引入

## 2. 当前基线

截至 `v1.2`，GoMap `secprobe` 已具备：

- `credential` / `unauthorized` 双类型探测
- Redis / MongoDB 未授权访问探测
- 成功 finding 的可选 enrichment
- `gomap weak` / `gomap port -weak` 双 CLI 入口
- 明确的默认保守策略

因此，后续版本不再需要回头解决“结果模型是否统一”这类底层问题，而可以直接在现有骨架上继续演进。

## 3. 后续演进原则

### 3.1 继续保持统一入口

后续仍建议保持：

- 对外统一 `pkg/secprobe`
- CLI 继续以：
  - `gomap weak`
  - `gomap port -weak`
  作为主入口

不建议现在重新拆成 GoMap 版 `servicebrute` / `hostdetection` 双模块。

### 3.2 默认行为继续保守

后续新增能力时，继续保持：

- 默认关闭
- 显式开启
- 默认最小必要验证
- 不做高风险写操作验证

### 3.3 先修扩展模式，再扩大协议面

`v1.2` 和 `v1.3` 已经证明基础架构与单目标链路是可行的。下一阶段更有价值的是先把 `secprobe` 的协议扩展模式整理清楚，而不是立刻把协议面铺得很大。

### 3.4 引擎外部集成能力放到后阶段

暂不建议提前引入：

- Temporal
- Redis 缓冲
- 分布式任务编排
- 大规模状态存储

这些更适合放到 `v2.0` 再评估。

## 4. v1.3：单目标流程稳定性与结果表达增强

### 4.1 目标

让 `secprobe` 从“功能已具备”升级到“单目标流程更稳定、探测结果更准确、更适合真实认证探测使用”。

### 4.2 建议范围

#### 4.2.1 单目标执行稳定性

建议补齐：

- 目标级熔断
- 更细粒度 stop policy
- 重试与失败分类边界
- 单目标执行链可观测性补强

建议重点控制的问题：

- 单目标链路中的误判
- 协议确认逻辑不稳定
- 某类协议单目标过度尝试

#### 4.2.2 结果表达增强

建议补齐：

- 更结构化的 `Evidence`
- enrichment 数据裁剪规则统一化
- finding 风险等级字段
- 只读 / 可写 / 可枚举 等能力表达
- 更明确的 skip / error reason

#### 4.2.3 CLI 控制面收敛

`v1.3` 不建议继续堆很多显式控制选项，而应优先保持：

- 默认行为保守
- 开关语义直观
- 少量高价值参数即可覆盖主场景
- 尽量复用现有参数语义，避免理解成本持续上升

如果确实需要补充控制项，也建议只增加与单目标稳定性直接相关、且用户容易理解的少量参数，而不是扩成一组复杂控制面。

### 4.3 v1.3 建议不做的事

`v1.3` 不建议做：

- 一次性接很多未授权协议
- 多目标并发治理
- 批量喷洒防护
- 全局速率限制
- 过多 CLI 控制选项
- 引入复杂任务编排
- 恢复历史上的结果回灌到 `assetprobe`
- 引入外部大爆破引擎

### 4.4 v1.3 完成标准

- 单目标流程更稳定
- finding 结果表达更稳定
- CLI 入口保持易用，且不会明显增加用户理解成本
- 不破坏 `v1.2` 已形成的清晰边界

## 5. v1.4：secprobe 扩展模式整改版

### 5.1 目标

让 `secprobe` 从“已经能继续加协议”升级成“能稳定、低成本、可持续扩协议”的模式。

`v1.4` 的重点不是多接几个协议，而是先把后续新增协议的工程模式整理清楚。

### 5.2 建议范围

#### 5.2.1 协议装配层整改

建议补齐：

- 把默认注册表与执行编排解耦
- 将协议装配逻辑从 `run.go` 中拆分出去
- 明确 registry / default registry / execution chain 的边界

目标是让：

- 执行编排更聚焦
- 协议注册更集中
- 后续加协议时改动面更可控

#### 5.2.2 协议目录与能力骨架统一

建议统一约束每个协议的扩展骨架，例如：

- `credential prober`
- `unauthorized prober`
- `enrichment`
- 协议测试文件

并明确：

- 一个协议一个目录
- 每种能力独立实现
- 不把 enrichment 主逻辑硬塞进 probe 主流程

#### 5.2.3 协议能力声明与可配置元数据收敛

建议逐步收敛哪些内容适合配置化，哪些必须代码实现。

优先考虑抽离为协议元数据的内容：

- 协议别名
- 默认端口
- 默认字典路径
- 是否支持 `credential` / `unauthorized` / `enrichment`

继续保留为代码实现的内容：

- 协议握手
- 认证流程
- 未授权确认动作
- enrichment 补采逻辑
- 失败分类

#### 5.2.4 协议扩展开发指南与测试模板

建议补齐：

- 协议扩展开发指南
- 新协议接入 checklist
- 协议测试模板
- 结果语义填写约束

重点统一：

- 如何确认成功
- 如何分类失败
- 如何填充 `Stage` / `FailureReason` / `Capabilities`
- 新增协议至少要覆盖哪些测试场景

### 5.3 v1.4 不建议做的事

`v1.4` 不建议做：

- 一次性扩一批新协议
- 做成纯配置驱动协议扩展
- 引入平台控制层概念
- 同时推进多目标治理
- 顺手增加过多 CLI 参数

### 5.4 v1.4 完成标准

- `secprobe` 的协议装配层边界更清晰
- 新协议扩展路径更标准化
- 协议能力与配置边界更明确
- 至少形成一份可执行的扩展开发规范
- 不破坏当前作为引擎端供外部平台集成的定位

## 6. 后续协议扩展阶段（可选）

在 `v1.4` 完成扩展模式整改后，如确有业务需求，再进入协议扩展阶段会更稳。

### 6.1 建议优先协议

可考虑按价值排序扩展：

1. Elasticsearch unauthorized
2. Memcached unauthorized
3. Zookeeper unauthorized
4. Kafka metadata anonymous access

### 6.2 设计要求

扩协议时继续遵守：

- 统一 `ProbeKind`
- 统一 finding 模型
- 统一确认逻辑要求
- 统一失败分类语义
- 不在 probe 内部硬塞 enrichment 主逻辑

## 7. v2.0：引擎契约稳定化与扩展层

### 7.1 目标

让 `secprobe` 从“本地可用的协议安全探测层”升级成“可被外部系统、批量任务和自动化流程稳定复用的引擎能力层”。

### 7.2 建议范围

#### 7.2.1 输出层与审计层

建议支持：

- 更稳定的 JSON 输出契约
- finding 审计字段
- 协议探测元数据
- enrichment 元数据
- 跳过原因和失败原因结构化

#### 7.2.2 插件式协议扩展

建议逐步形成：

- 探测器注册约定
- enrichment 注册约定
- 协议测试模板
- 结果证据模板

这会让后续扩协议更稳定，而不是继续靠散点实现。

#### 7.2.3 更适合外部系统集成的控制层

如果未来确实存在更强的外部集成诉求，可以在 `v2.0` 再评估：

- 多目标并发治理
- 批量喷洒防护
- 全局速率限制
- 批量任务保护策略
- 任务编排
- 批量调度
- 外部存储
- 审计归档
- 风险聚合

### 7.3 v2.0 仍不建议盲目引入的内容

即便到 `v2.0`，也不建议为了“看起来更像平台底座”而直接引：

- 与现有项目无强需求绑定的 Temporal
- 过早的 Redis 缓冲层
- 大而全却低测试覆盖的协议爆破体系
- 复杂但没有明确消费方的数据模型

## 8. 建议优先级

如果后续继续推进，推荐顺序是：

1. `v1.3`
   - 单目标流程稳定性
   - 结果表达增强
   - CLI 控制面补强
2. `v1.4`
   - `secprobe` 扩展模式整改
   - 协议装配层拆分
   - 扩展开发规范收口
3. 后续协议扩展阶段（如确有必要）
   - 按优先级逐个扩协议
4. `v2.0`
   - 稳定输出契约
   - 插件化扩展层
   - 审计与复用能力
   - 多目标治理能力再评估

## 9. 当前不建议提前做的事项

以下能力目前建议继续放在 roadmap 中，但不要现在提前实现：

- 外部爆破引擎接入
- 写操作验证
- 大范围协议接入
- 资产结果回灌 `assetprobe`
- 重构成双模块体系
- 分布式任务编排

## 10. 对团队协作的建议

后续继续开发时，建议保持当前做法：

- 一阶段一 spec
- 一阶段一 implementation plan
- 分提交推进
- 每个任务都跑 review + verification

这样做的好处是：

- 需求边界不容易漂
- 代码审查更容易聚焦
- 回滚和定位更简单

## 11. 结论

`v1.2` 已经把 `secprobe` 的底层架构问题基本解决了。

因此下一阶段真正值得投入的方向不是“再堆几个协议”，而是：

- 让运行更稳
- 让结果更清晰
- 让扩展模式更清晰
- 让输出更适合被外部消费

建议把 `v1.3` 定位为“单目标流程稳定性与结果表达增强版”，把 `v1.4` 定位为“`secprobe` 扩展模式整改版”，把 `v2.0` 定位为“引擎契约稳定化与扩展层版”。

这样的顺序更符合 GoMap 当前作为引擎端能力组件的定位，也更符合“先修模式，再扩协议，再稳定契约”的实际推进节奏。
